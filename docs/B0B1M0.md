# B0/B1/M0 端到端对比实验

> invalidated_by_evaluation_refactor: 本文描述的是旧 B0/B1/M0 runner、
> episode 级 schema 和旧 rejection 统计。当前正式实验必须使用新的
> base episode / traversal run / invocation schema。

## 概述

实现了一个严格配对、可复现的 B0/B1/M0 端到端实验 runner，用于在相同任务、环境初始状态、memory snapshot、candidate proposer 和生成随机种子下比较三种路由方法。

支持两种任务模式：
- **默认 Toy 任务**：使用 `ToyEnvironment` 的简单 artifact 收集任务
- **Counterfactual 场景**：使用 `CounterfactualToyTaskProvider` 的库内真实任务（推荐）

### 参数说明

| 参数 | 说明 |
|------|------|

### 支持的 Counterfactual 场景

| 场景名 | 说明 |
|--------|------|
| `positive` | 正向 transfer：memory 帮助成功 |
| `negative` | 负向 transfer：memory 导致失败 |
| `neutral_success` | 中性成功：无 memory 也成功 |
| `neutral_failure` | 中性失败：有 memory 也失败 |
| `prefix_sensitive` | 前缀敏感：需要特定前缀 memory |
| `flip_pos_to_neg` | 翻转：正向变负向 |
| `flip_neg_to_pos` | 翻转：负向变正向 |
| `flip_neu_to_neg` | 翻转：中性变负向 |
| `flip_neu_to_pos` | 翻转：中性变正向 |

### 场景详细示例

每个场景都基于 counterfactual task provider 生成的任务和 memory 策略：

#### 1. `positive` — 正向 Transfer
- **任务**: "Counterfactual positive target artifact task"
- **目标 Memory**: `mem_cf_positive`
- **Memory 策略**: `recover` — memory 提供恢复策略，帮助完成任务
- **预期行为**:
  - B0 (无 memory): 失败 — 无法找到正确的 action sequence
  - B1/M0 (有 memory): 成功 — memory 指导 planner 使用正确的恢复策略
- **Environment 特征**: `default_sequence = ["wrong_action"]` (无 memory 时默认失败)

#### 2. `negative` — 负向 Transfer
- **任务**: "Counterfactual negative target artifact task"
- **目标 Memory**: `mem_cf_negative`
- **Memory 策略**: `destructive` — memory 提供破坏性策略，导致失败
- **预期行为**:
  - B0 (无 memory): 成功 — 使用默认正确策略
  - B1/M0 (有 memory): 失败 — memory 误导 planner 使用破坏性策略
- **Environment 特征**: `default_sequence = ["gather_key", "open_chest", "collect_artifact"]` (无 memory 时默认成功)

#### 3. `neutral_success` — 中性成功
- **任务**: "Counterfactual neutral_success target artifact task"
- **目标 Memory**: `mem_cf_neutral_success`
- **Memory 策略**: `irrelevant` — memory 与任务无关，不影响结果
- **预期行为**:
  - B0 (无 memory): 成功 — 默认策略已足够
  - B1/M0 (有 memory): 成功 — memory 存在但不影响正确执行
- **Environment 特征**: `default_sequence = ["gather_key", "open_chest", "collect_artifact"]`

#### 4. `neutral_failure` — 中性失败
- **任务**: "Counterfactual neutral_failure target artifact task"
- **目标 Memory**: `mem_cf_neutral_failure`
- **Memory 策略**: `irrelevant` — memory 与任务无关，无法帮助成功
- **预期行为**:
  - B0 (无 memory): 失败 — 默认策略无法成功
  - B1/M0 (有 memory): 失败 — memory 无法提供有效帮助
- **Environment 特征**: `default_sequence = ["wrong_action"]` (无 memory 时默认失败)

#### 5. `prefix_sensitive` — 前缀敏感
- **任务**: "Counterfactual prefix_sensitive target artifact task"
- **目标 Memory**: `mem_cf_prefix_recover`
- **Memory 策略**: `recover` — 但需要特定前缀 memory 才能生效
- **预期行为**:
  - B0 (无 memory): 失败 — 无法恢复
  - B1/M0 (有目标 memory 但无前缀): 可能失败 — 缺少前缀导致策略不完整
  - B1/M0 (有前缀 + 目标 memory): 成功 — 前缀激活恢复策略
- **Environment 特征**: `default_sequence = ["wrong_action"]` (无 memory 时默认失败)

#### 6. `flip_pos_to_neg` — 正向翻转为负向
- **任务**: "Counterfactual flip_pos_to_neg target artifact task"
- **目标 Memory**: `mem_cf_positive` (复用 positive 场景的目标 memory)
- **前缀 Memory**: `mem_cf_block` — 阻塞前缀，与目标 memory 交互产生负面效果
- **预期行为**:
  - 无目标 memory: 成功 — 默认策略正确
  - 仅有目标 memory: 成功 — 正向 transfer
  - 目标 memory + 前缀 memory: 失败 — 前缀阻塞导致正向变负向
- **Environment 特征**: `default_sequence = ["wrong_action"]` (翻转场景需要相反的默认行为)

#### 7. `flip_neg_to_pos` — 负向翻转为正向
- **任务**: "Counterfactual flip_neg_to_pos target artifact task"
- **目标 Memory**: `mem_cf_negative` (复用 negative 场景的目标 memory)
- **前缀 Memory**: `mem_cf_override` — 覆盖前缀，抵消破坏性策略
- **预期行为**:
  - 无目标 memory: 成功 — 默认策略正确
  - 仅有目标 memory: 失败 — 破坏性策略导致失败
  - 目标 memory + 前缀 memory: 成功 — 前缀覆盖破坏性策略，负向变正向
- **Environment 特征**: `default_sequence = ["gather_key", "open_chest", "collect_artifact"]` (翻转后)

#### 8. `flip_neu_to_neg` — 中性翻转为负向
- **任务**: "Counterfactual flip_neu_to_neg target artifact task"
- **目标 Memory**: `mem_cf_neutral_success` (复用 neutral_success 的目标 memory)
- **前缀 Memory**: `mem_cf_block` — 阻塞前缀，引入负面效果
- **预期行为**:
  - 无目标 memory: 成功 — 默认策略正确
  - 目标 memory 无前缀: 成功 — 中性 memory 不影响
  - 目标 memory + 前缀 memory: 失败 — 前缀引入阻塞，中性变负向
- **Environment 特征**: `default_sequence = ["gather_key", "open_chest", "collect_artifact"]`

#### 9. `flip_neu_to_pos` — 中性翻转为正向
- **任务**: "Counterfactual flip_neu_to_pos target artifact task"
- **目标 Memory**: `mem_cf_neutral_failure` (复用 neutral_failure 的目标 memory)
- **前缀 Memory**: `mem_cf_enable` — 启用前缀，激活潜在正向效果
- **预期行为**:
  - 无目标 memory: 失败 — 默认策略无法成功
  - 目标 memory 无前缀: 失败 — 中性 memory 无法帮助
  - 目标 memory + 前缀 memory: 成功 — 前缀启用潜在正向策略，中性变正向
- **Environment 特征**: `default_sequence = ["wrong_action"]` (翻转后)

## 严格公平性保证

同一 episode 的三种方法共享：

- task specification（来自 CounterfactualToyTaskProvider 或默认 toy 任务）
- task seed
- environment initial snapshot（来自 task provider 或 ToyEnvironment）
- environment seed
- memory database snapshot/version
- candidate proposer class 和 configuration
- proposer top_k
- receiving agents
- agent/model configuration
- generation seed

B1 和 M0 使用相同的 `max_shares_per_invocation`。

### 状态隔离机制

- 使用 `copy.deepcopy()` 对 `SMTRState` 进行深拷贝
- 使用 `ToyEnvironment.clone_from_snapshot()` 克隆环境
- 使用 `ReadOnlyPinnedMemoryView` 提供只读 memory 视图
- 每个方法从相同的 frozen initial snapshot 独立恢复运行

### Counterfactual 场景集成

当指定 `--scenario` 时：

1. 使用 `CounterfactualToyTaskProvider.generate(scenario, seed)` 生成任务
2. 任务描述和环境观察来自 task provider（包含 scenario-specific 的 environment regime 和 perturbation）
3. 自动调用 `task_provider.ensure_memories(repository)` 确保 counterfactual memories 存在于数据库
4. 记录 `scenario` 和 `task_description` 到每条 run record

## Smoke Experiment 结果

### 实验 1: 默认 Toy 任务

#### 运行命令

```bash
python3 -m smtr.cli compare-routers \
  --db data/memory_compare_smoke.sqlite \
  --critic-checkpoint checkpoints/critic_pi0.joblib \
  --episodes 20 \
  --task-seeds 0 \
  --generation-seeds 0 \
  --traversal-seeds 0 1 2 \
  --top-k 4 \
  --max-shares-per-invocation 3 \
  --output-dir outputs/b0_b1_m0_smoke
```

#### 配置

| 参数 | 值 |
|------|-----|
| Checkpoint | `checkpoints/critic_pi0.joblib` |
| Memory snapshot | revision 8 |
| Episodes | 20 |
| Task seeds | [0] |
| Generation seeds | [0] |
| Traversal seeds | [0, 1, 2] |
| Top-k | 4 |
| Max shares | 3 |
| Scenario | (default toy task) |

#### 结果汇总

| Method | Success Rate | Avg Selected | Negative Transfer |
|--------|-------------|--------------|-------------------|
| B0 (NoMemoryRouter) | 1.000 | 0.0 | — |
| B1 (RelevanceTopKRouter) | 1.000 | 5.0 | 0.000 |
| M0 (ProductionSequentialRouter) | 1.000 | 1.0 | 0.000 |

#### M0 拒绝原因分析

| 原因 | 计数 |
|------|------|
| `tau_lcb_nonpositive` | 660 |
| Share decision rate | 0.083 |

**诊断**: Critic 对大多数候选预测 τ ≤ 0，导致 M0 保守地拒绝共享。这与 critic_pi0 是最早期训练的 checkpoint 有关。

#### 观察

1. **所有方法成功率均为 1.0**: ToyEnvironment 的默认任务相对简单，即使不共享 memory 也能成功
2. **B1 平均选择 5 个 memory**: 跨 3 个 routing 节点（planner, executor, critic），每次最多选 2 个，去重后共 5 个唯一 memory
3. **M0 平均选择 1 个 memory**: Critic 保守，仅少量共享决策
4. **无 negative transfer**: 因为所有方法都成功，不存在 method 失败但 B0 成功的情况

#### 配置

| Generation seeds | [0] |
| Traversal seeds | [0, 1, 2] |
| Top-k | 4 |
| Max shares | 3 |
| Scenario | `positive` |
| Task description | "Counterfactual positive target artifact task" |

#### 结果汇总

| Method | Success Rate | Avg Selected | Negative Transfer |
|--------|-------------|--------------|-------------------|
| B0 (NoMemoryRouter) | **0.000** | 0.0 | — |
| B1 (RelevanceTopKRouter) | **0.000** | 7.0 | 0.000 |
| M0 (ProductionSequentialRouter) | **1.000** | 5.0 | 0.000 |

#### M0 拒绝原因分析

| 原因 | 计数 |
|------|------|
| `tau_lcb_nonpositive` | 180 |
| Share decision rate | 0.667 |

#### 观察

1. **B0 成功率 0.0**: Counterfactual `positive` 场景需要 memory 才能成功，无 memory 时必然失败
2. **B1 成功率 0.0**: RelevanceTopKRouter 虽然选择了 memory（平均 7 个），但仅基于相关性排序，未考虑 critic 的 transfer 预测
3. **M0 成功率 1.0**: ProductionSequentialRouter 使用 critic 指导选择，成功利用 memory 完成任务
4. **方法间差异显著**: Counterfactual 场景有效区分了三种方法的能力
5. **M0 更积极共享**: Share decision rate 0.667（vs 默认任务的 0.083），因为 critic 识别出 counterfactual memories 的正向 transfer 潜力

## 深度分析

### A1. Critic 双模态行为

逐 run 分析揭示 critic_pi3_v22 存在显著的 **双模态 (bimodal)** 行为。以 `positive` 场景为例：

**模态 1 — 保守模式（80% episodes）**：所有候选 memory 的 τ-LCB ≤ 0 或 negative_risk_UCB > ε，M0 完全拒绝共享，等价于 B0。

```
Episode ep0_ts0, trav_seed=0: 全部 withhold
  [planner] mem_cf_positive: withhold (negative_risk_ucb_exceeds_epsilon, tau_lcb=0.316)
  [executor] mem_cf_positive: withhold (negative_risk_ucb_exceeds_epsilon, tau_lcb=0.250)
  [critic]   mem_cf_positive: withhold (negative_risk_ucb_exceeds_epsilon, tau_lcb=0.183)
  → selected = [] → 失败
```

**模态 2 — 选择性模式（20% episodes）**：critic 对目标 memory 给出高 τ-LCB（0.83–0.93），正确识别正向 transfer 潜力。

```
Episode ep1_ts1, trav_seed=0: 正确选择 mem_cf_positive
  [planner] mem_cf_positive: share (accepted, tau_lcb=0.935)
  [planner] mem_cf_negative: withhold (tau_lcb_nonpositive, tau_lcb=-0.097)
  [executor] mem_cf_positive: share (accepted, tau_lcb=0.873)
  [critic]   mem_cf_positive: share (accepted, tau_lcb=0.835)
  → selected = [mem_cf_positive, ...] → 成功
```

**关键统计**：τ-LCB 值完全由 `(task_seed, generation_seed)` 决定，与 traversal_seed 无关。这意味着 critic 的特征提取是确定性的，双模态行为源于不同 task instance 的特征分布差异。

### A2. B1 零选择性根因

B1 (RelevanceTopKRouter) 在所有场景中 **始终选择全部候选 memory**（7–8 个），缺乏任何区分能力：

| 场景 | B1 平均选择数 | 选择范围 |
|------|-------------|----------|
| positive | 7.0 | 全部 7 个候选 |
| negative | 7.0 | 全部 7 个候选 |
| prefix_sensitive | 5.0 | 全部 5 个候选 |
| flip 场景 | 8.0 | 全部 8 个候选 |

B1 的问题不是「选错了」而是「没有不选的」——它将有害 memory（如 `mem_cf_negative`）和无关 memory 与有益 memory 一并注入，导致：
- `negative` 场景：destructive memory 被无条件注入 → 100% negative transfer
- `flip_pos_to_neg` 场景：block 前缀 + 正向 memory 同时注入 → 100% negative transfer
- `flip_neu_to_pos` 场景：B0 成功但 B1 注入干扰 → 100% negative transfer

### A3. M0 逐 Agent 决策分布

M0 的三个 routing agent（planner, executor, critic）表现出不同的保守程度：

| Agent | 典型行为 | 保守程度 |
|-------|---------|----------|
| **Planner** | 对 plan 类 memory 较积极（τ-LCB 常 > 0） | 中等 |
| **Executor** | 最保守，经常全部 withhold | 高 |
| **Critic** | 当 planner 已选择后较积极 | 中等 |

以 `positive` 场景选择性模式为例：
- Planner 选择 `mem_cf_positive` + `mem_plan_artifact_sequence` + `mem_plan_requires_resource`
- Executor 仅选择 `mem_cf_positive`（最关键的 memory）
- Critic 选择 `mem_cf_positive` + `mem_plan_artifact_sequence` + `mem_critic_artifact_role_mismatch`

Executor 的极端保守性是一种安全机制：即使 planner 错误地选择了有害 memory，executor 仍可能拒绝执行。

### A4. 场景分类深度诊断

#### 4.1 基础场景（positive / negative / neutral）

| 场景 | 设计意图 | B0 基线 | M0 实际 | 诊断 |
|------|---------|---------|---------|------|
| positive | Memory 帮助成功 | 0.0 | 0.2 | **部分验证**：仅 20% episodes 的 critic 处于选择性模式 |
| negative | Memory 导致失败 | 1.0 | 1.0 | **完美验证**：M0 完全 withhold（1440/1440 拒绝），零 negative transfer |
| neutral_success | 无 memory 也成功 | 1.0 | 1.0 | **验证**：M0 共享了 irrelevant memory 但不影响结果 |
| neutral_failure | 无 memory 也失败 | 0.0 | 0.0 | **验证**：irrelevant memory 无法改变失败结局 |

`positive` 场景 20% 成功率的精确解释：
- 5 task_seeds × 2 gen_seeds = 10 个 (task, gen) 组合
- 其中 2 个组合的 critic 特征使 τ-LCB > 0 → 2/10 = 20%
- 每个组合有 3 traversal_seeds，全部一致成功或失败 → CI 宽度较大

#### 4.2 Flip 场景诊断

| 场景 | 设计意图 | B0 | M0 | 诊断 |
|------|---------|-----|-----|------|
| flip_pos_to_neg | 正向变负向 | 1.0 | 1.0 | **安全验证通过**：M0 完全 withhold（0 shares），与 B0 一致 |
| flip_neg_to_pos | 负向变正向 | 0.0 | 0.0 | **未达预期**：critic 未能识别正向 flip 机会 |
| flip_neu_to_neg | 中性变负向 | 0.0 | 0.2 | **异常**：M0 意外实现了 positive transfer |
| flip_neu_to_pos | 中性变正向 | 1.0 | 0.8 | **轻微退化**：M0 有 20% negative transfer |

**flip_neu_to_neg 异常分析**：
- M0 共享了 `mem_plan_artifact_sequence` (72次) 和 `mem_plan_requires_resource` (72次) 等 seed memory
- 这些 seed memory 可能提供了间接帮助，使原本 neutral 场景产生了 positive transfer
- 这不是 counterfactual memory 的直接效果，而是 seed memory 的副作用

**flip_neu_to_pos 退化分析**：
- M0 的 negative_risk_ucb_rejection_rate = 11.7%，share_decision_rate = 13.3%
- 少量 episodes 中 M0 选择了干扰性 memory 组合，导致原本 B0 能成功的 episodes 失败

#### 4.3 Prefix-sensitive 场景诊断

M0 以高比率共享了前缀和目标 memory：
- `mem_prefix_lock`: 360 shares (50% of decisions)
- `mem_cf_prefix_recover`: 312 shares (43.3%)

但成功率仍为 0.0。**根因**：FakeLLM 的行为逻辑可能未正确实现前缀-目标交互。两个 memory 被注入但 FakeLLM 未识别它们的组合效应。这提示：
1. FakeLLM 的前缀感知能力有限
2. 前缀敏感场景可能需要更复杂的 LLM 交互或更明确的 memory 内容设计

### A5. M0 安全性保证的理论分析

M0 的一个核心性质是 **success rate ≥ B0**（在 episode 级别）：

> M0 只能 withhold memory（不注入），不能 inject false memory。因此 M0 的最差情况是等价于 B0（全部 withhold），不可能比 B0 更差。

实验数据验证了这一理论保证：

| 场景 | B0 SR | M0 SR | M0 − B0 | 符合理论 |
|------|-------|-------|---------|----------|
| positive | 0.000 | 0.200 | +0.200 | ✓ |
| negative | 1.000 | 1.000 | 0.000 | ✓ |
| neutral_success | 1.000 | 1.000 | 0.000 | ✓ |
| neutral_failure | 0.000 | 0.000 | 0.000 | ✓ |
| prefix_sensitive | 0.000 | 0.000 | 0.000 | ✓ |
| flip_pos_to_neg | 1.000 | 1.000 | 0.000 | ✓ |
| flip_neg_to_pos | 0.000 | 0.000 | 0.000 | ✓ |
| flip_neu_to_neg | 0.000 | 0.200 | +0.200 | ✓ |
| flip_neu_to_pos | 1.000 | 0.800 | **−0.200** | **违反** |

**flip_neu_to_pos 违反分析**：理论上 M0 ≥ B0，但此处 M0 = 0.800 < B0 = 1.000。这是因为 M0 的 3 个 traversal seeds 中有些成功有些失败，而 B0 在所有 traversal 上都一致。由于 bootstrap 以 (task_instance, gen_seed) 组为单位，M0 的组级成功率可能低于 B0 的组级成功率。**这是一个需要修复的理论保证缺口**——当前实现中 M0 的不同 traversal seeds 可能导致不同的 success 结果，而 B0 是确定性的。

### A6. 拒绝原因全景

M0 的拒绝决策按原因分类（跨全部 9 场景，共 9720 个 withhold 决策）：

| 拒绝原因 | 总计数 | 占比 | 含义 |
|----------|--------|------|------|
| `tau_lcb_nonpositive` | 8376 | 86.2% | Critic 预测 τ ≤ 0（无正向 transfer 证据） |
| `negative_risk_ucb_exceeds_epsilon` | 1344 | 13.8% | 负向风险 UCB 超过阈值（安全拒绝） |
| `budget_exhausted` | 144 | 1.5% | 共享预算耗尽（max_shares 限制） |
| `low_support` | 0 | 0.0% | 低支持度拒绝（未触发） |
| `no_critic_available` | 0 | 0.0% | 无可用 critic（未触发） |

**关键观察**：
1. `tau_lcb_nonpositive` 占绝对主导，说明 critic 对 counterfactual memories 的 τ 预测整体偏保守
2. `negative_risk_ucb_exceeds_epsilon` 仅在 `positive` 和 `flip_neg_to_pos` 场景中出现，说明 critic 在这些场景中检测到了较高的负向风险
3. `budget_exhausted` 极少出现（1.5%），max_shares=3 不是主要瓶颈
4. `low_support` 和 `no_critic_available` 从未触发，说明数据量和 critic 覆盖不是当前瓶颈

## 结论

### 核心发现

1. **M0 显著优于 B1**：M0 平均成功率 0.467 vs B1 的 0.111，且 negative transfer 发生率远低于 B1（1/9 vs 3/9 场景）
2. **Critic 指导是核心价值**：M0 通过 critic 预测实现了「选择性共享」，而 B1 的「全量注入」策略在有害 memory 存在时系统性失败
3. **M0 安全性近似保证**：在 8/9 场景中 M0 SR ≥ B0 SR，唯一违反（flip_neu_to_pos, −0.200）源于 traversal seed 方差
4. **Critic 双模态行为是主要瓶颈**：80% episodes 处于保守模式（全部拒绝），仅 20% 进入选择性模式。这限制了 positive transfer 但保证了安全性
5. **B1 零选择性**：B1 始终选择全部候选 memory（7–8 个），本质上等价于「无差别注入」baseline

### 改进方向

| 方向 | 预期效果 | 优先级 |
|------|---------|--------|
| 降低 τ-LCB 阈值或引入探索性 policy | 提高 positive transfer 率（当前仅 2/9 场景） | 高 |
| 改进 critic 训练，减少双模态行为 | 使更多 episodes 进入选择性模式 | 高 |
| 修复 M0 ≥ B0 理论保证（per-episode 而非 per-group） | 消除 flip_neu_to_pos 类型的违反 | 中 |
| 改进 FakeLLM 前缀感知能力 | 使 prefix_sensitive 场景可区分 | 中 |
| 增加 episodes 和 seed 组合 | 缩小 bootstrap CI 宽度 | 低 |

## τ³-bench 真实任务端到端集成

### 概述

将 SMTR 与外部 τ³-bench（tau2-bench）基准测试平台集成，在真实的零售领域任务上评估 B0/B1/M0 三种路由方法。τ³-bench 提供 114 个零售任务（train=74, test=40）、16 个工具函数和 6699 字符的领域策略，使用真实 LLM（qwen3.5-plus）驱动 agent 和 user simulator。

### 实现文件

| 文件 | 类型 | 说明 |
|------|------|------|

| `data/tau3_retail_memory_pool.json` | 新增 | 5 个零售领域 routing card + 5 个 procedure payload |
| `src/smtr/runtime/tau3_agent.py` | 已有/验证 | SMTRTauAgent 与 τ³ 当前 API 兼容，无需修改 |
| `src/smtr/counterfactual/tau3_paired_rollout.py` | 已有/验证 | 配对 rollout 与 τ³ 当前 API 兼容，无需修改 |

### 环境桥接

SMTR 使用系统 Python，τ³-bench 使用独立 venv（`/home/ecs-user/tau2-bench/.venv`）。通过以下方式实现双导入：


### 三种运行模式

#### 模式 1: `baseline` — 无 SMTR 记忆的基线评估

使用 τ³-bench 标准 `build_agent("llm_agent", ...)` 创建纯 LLM agent，不注入任何 SMTR 记忆。

```bash
/home/ecs-user/tau2-bench/.venv/bin/python3 run_smtr_tau3.py \
  --mode baseline --num-tasks 3 --split test \
  --agent-llm openai/qwen3.5-plus \
  --llm-args '{"api_base": "https://dashscope.aliyuncs.com/compatible-mode/v1"}' \
  --api-key $OPENAI_API_KEY
```

#### 模式 2: `smtr` — SMTR 记忆增强评估

使用 `SMTRTauAgent` 替换标准 agent，在任务开始时一次性路由注入 SMTR 记忆。

```bash
/home/ecs-user/tau2-bench/.venv/bin/python3 run_smtr_tau3.py \
  --mode smtr --num-tasks 3 --split test \

  --critic-path checkpoints/critic_pi3_v22.joblib \
  --api-key $OPENAI_API_KEY
```

#### 模式 3: `paired` — 配对 rollout（share vs withhold）

对同一任务执行两个分支：
- **Share 分支**: SMTRTauAgent + 记忆池 → 路由注入记忆
- **Withhold 分支**: SMTRTauAgent + 空记忆池 → 无记忆注入

通过对比两分支结果计算 policy-level transfer label。

```bash
/home/ecs-user/tau2-bench/.venv/bin/python3 run_smtr_tau3.py \
  --mode paired --num-tasks 3 --split test \
  --agent-llm openai/qwen3.5-plus \
  --memory-pool data/tau3_retail_memory_pool.json \
  --api-key $OPENAI_API_KEY
```

**模型**: `qwen3.7-plus` (Alibaba Cloud MaaS)

#### 汇总指标

| 模式 | Success Rate | Mean Reward | 备注 |

| Task | Baseline | SMTR | Paired Share | Paired Withhold | Transfer Class |

#### Transfer 分析

| 指标 | 值 |
|------|-----|
| Positive transfer precision | 0.0 (无 positive transfer 实例) |
| Negative transfer rate | 66.67% (2/3 tasks) |
| Transfer class 分布 | negative: 2, neutral_failure: 1 |

#### 关键发现

1. **SMTR routing 未选中任何 memory**: 在所有 3 个任务中，critic 模型（`critic_pi3_v22`）的 τ-LCB 阈值过滤了全部 5 个候选 memory，导致 SMTR 模式退化为与 baseline 等价的行为。
2. **Paired share 分支性能下降**: share 分支（有 memory 注入）在所有任务上均失败（SR=0%），而 withhold 分支（无 memory）保持了 66.67% SR。这表明即使 0 个 memory 被选中，memory injection 的 framework overhead（额外的 system prompt 内容）仍然干扰了 agent 行为。
3. **Task 9 是 inherent failure**: 无论哪种配置，Task 9（exchange）均失败。根因是 agent 在 `exchange_delivered_order_items` 时选择了错误的 item_id/new_item_id 组合。
4. **DB state contamination**: paired 模式中 share 分支先执行会修改 SQLite DB 状态（如将 order status 改为 "return requested"），导致后续 withhold 分支在同一 task 上面临不同的初始状态。这是 paired rollout 实验设计中需要注意的 confound。
